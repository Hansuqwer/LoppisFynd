  <action>
Harden `tradera-proxy` with a deliberate protection posture and a stable error response shape.

Implementation requirements:
- Refactor to match the testable pattern used by `supabase/functions/cloud-ai-proxy/index.ts`:
  - Export `handleRequest(req: Request, deps?: { env?: { get(k): string|undefined }, fetch?: FetchLike, rateLimit?: (key: string) => Promise<{ allowed: boolean; retryAfterSeconds?: number }> })`.
  - Keep `Deno.serve((req) => handleRequest(req))` for runtime.
- Add a hard body-size guard similar in spirit to cloud-ai-proxy (keep it small; request is tiny).
- Preserve CORS behavior.
- Add `cache-control: no-store` to all JSON responses (match cloud-ai-proxy).

Protection:
- Implement durable rate limiting using Upstash Redis + `@upstash/ratelimit`:
  - Use env vars `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`.
  - Rate limit key should prefer a stable user identifier when available; otherwise fall back to IP:
    - Use `x-forwarded-for` (first IP) when present; otherwise `cf-connecting-ip`; otherwise `unknown`.
  - Suggested policy (reasonable defaults, document in code): e.g. `10 requests / minute / key`.
- If rate limited, return HTTP 429 with:
  - JSON body containing a stable error object
  - `retry-after` header when `retryAfterSeconds` is known

Error contract (apply to all non-2xx responses):
- Return `{ error: { code: string, message: string, retryAfterSeconds?: number } }`.
- Codes to use at minimum:
  - `method_not_allowed` (405)
  - `invalid_json` (400)
  - `invalid_request` (400)
  - `rate_limited` (429)
  - `server_not_configured` (500)
  - `upstream_failed` (502)
  - `parse_failed` (502)

Tests:
- Add `supabase/functions/tradera-proxy/tests/index_test.ts` covering:
  - 405 method not allowed returns the error object shape.
  - 400 invalid JSON returns stable code.
  - Injected `rateLimit` that denies returns 429 + error code + retryAfterSeconds.
  - Success path still returns a JSON object compatible with existing `TraderaProxyResponse`.
  </action>

  - Avoid a tight retry loop; keep total attempts bounded.
- Keep existing retry behavior for transient failures (timeouts, socket errors, 5xx), but ensure the final thrown error is concise (avoid dumping huge response bodies).
